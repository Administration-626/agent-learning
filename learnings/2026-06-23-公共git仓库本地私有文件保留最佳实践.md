# 公共Git仓库本地私有文件保留最佳实践

Date: 2026-06-23
Agent: Antigravity

## 背景 (Background)
在多人协作的公共 Git 仓库中，开发者常常需要保留一些“本地特供”的未提交文件（例如本地测试脚本、临时日志或特殊修改的配置等）。同时，公共仓库的 `.gitignore` 属于项目级别的共识约束，不应当为了满足个人特定环境的需求而随意修改。

## 问题 (Problem)
在频繁使用 `git pull` 同步上游更新的工作流中：
1. 随意新增的文件会使 `git status` 变得异常脏乱，容易引发误提交。
2. 针对已追踪公共文件的本地临时修改，容易引发拉取冲突。
3. 依赖 `git stash` 作为保留修改的常规手段是一种反模式，频繁地 `stash` / `pull` / `stash apply` 会带来极高的心智负担和意外合并风险。

## 解决方案 (Solution)
- **隐身新增私有文件**：使用仓库内的 `.git/info/exclude` 机制。其语法规则和 `.gitignore` 完全一致，可以将本地固定的临时工作目录（如 `local_scripts/`）直接忽略。该文件纯本地生效，绝不会被意外推送到远程。
- **屏蔽已被追踪文件的本地修改**：使用本地索引挂起机制 `git update-index --skip-worktree <file>`。执行后，Git 将彻底无视对该文件的本地修改。需要真实提交代码时，再用 `--no-skip-worktree` 恢复。

## 避坑指南 (Pitfalls)
- 严禁为了“保留长期使用的本地特供修改”而反复执行 `git stash` 循环。
- （事实/推论）：对于需要作为“常驻本地状态”的未提交变更，如果滥用 `stash pop`（弹出并丢弃记录）而非 `stash apply`（保留记录），在拉取最新远端代码发生合并失败时，很容易丢失这份没有 commit 记录的私有修改补丁。
