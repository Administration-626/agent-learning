# cargo install 与 crates.io 发布机制

Date: 2026-06-20
Agent: Gemini CLI

## Background
用户询问为何像 ActPlane 这种涉及底层系统调用拦截的 eBPF 工具，可以直接通过 `cargo install` 下载，以及这种第三方包是否允许放置在 Cargo 的仓库中。这反映了用户对包管理器的分发机制、预编译内核工具的分发方式以及运行权限隔离的好奇。

## Context / Problem
对于需要触碰操作系统内核（eBPF）的工具，传统的认知是需要复杂的本地内核头文件、LLVM/Clang 编译环境，并且通常由系统包管理器（如 apt, yum）通过官方源分发，以保证系统安全。而 `cargo install` 的简单性似乎与此相矛盾。

## Solution / Learnings

### 1. crates.io 的开放生态
Rust 的官方包仓库 `crates.io` 和 Node.js 的 `npm` 或 Python 的 `PyPI` 一样，是一个**完全开放的去中心化公共仓库**。任何人只要注册账号且包名未被占用，都可以通过 `cargo publish` 发布代码。官方不进行人工的安全性审查，因此第三方可以自由托管项目。

### 2. cargo install 的本质
`cargo install` 并不是下载预编译好的二进制文件（通常情况），而是**下载 Rust 源码并在本地编译**。整个下载和编译过程都在**普通用户（User-space）权限**下进行，没有任何特权。生成的可执行文件会被放置在 `~/.cargo/bin` 目录下。

### 3. CO-RE 技术（一次编译，到处运行）
ActPlane 等现代 eBPF 工具利用了 **CO-RE (Compile Once, Run Everywhere)** 技术。开发者在发布到 crates.io 之前，已经把复杂的 eBPF 内核 C 代码预先编译成了跨平台兼容的 `.bpf.o` 二进制块，**并将其直接作为静态资源内嵌到了 Rust 的源码外壳中**。
因此，当用户执行 `cargo install` 时，本地实际只编译了一层薄薄的 Rust 用户态外壳程序，完全避开了复杂的内核编译工具链。

### 4. 严格的权限隔离
系统的安全性不是靠“限制下载”来保证的，而是靠“限制运行”。
- **分发/安装阶段**：普通权限，任何人都可以把工具源码买回家并组装成二进制。
- **运行/生效阶段**：当执行 `sudo actplane run` 时，才真正涉及到权限跃迁。只有被授予了 `sudo`（或 `CAP_BPF` 特权），外壳程序才有权限把内置的 eBPF 探针注入到操作系统内核中发挥作用。

## Pitfalls
- **误区一**：认为能通过语言包管理器安装的都是“安全无害的业务逻辑代码”。实际上代码可以包含任何逻辑。
- **误区二**：认为 eBPF 工具一定需要繁杂的本地编译。利用 CO-RE 和源码内嵌，可以实现极度丝滑的分发体验。
- **误区三**：将分发渠道的开放性与系统的安全性混淆。分发可以是开放的（如 crates.io），但提权注入内核（sudo / capabilities）这一关由操作系统死死把守。
