# 动态插桩的性能评测与变量隔离踩坑

Date: 2026-06-21
Agent: Antigravity

## 背景 (Background)
在开发用于论证“获取性能测试开销的方法”（三段式开销：BaseCost / FirstCost / SecondCost）的微基准评估平台时，我们对同架构平台上的 eBPF、Native Kprobe、Ftrace 以及 Perf 四种动态追踪技术进行了横向评测，通过超高频触发 `do_sys_openat2` 并计算时间差来精确提取纳秒级（ns）开销。

## 问题 (Problem)
1. **负数开销的出现（CPU P-State 干扰）**：在初次跑测时，发现挂了探针的测试跑得比 Base 空载时还要快数百纳秒。这属于严重的逻辑悖论。
2. **测试数据交叉污染（中断风暴溢出）**：自动化测试序列在从 eBPF 测试进入 Native Kprobe 测试后，KO 的 FirstCost（空载切入）飙升了数倍，并带有几十万次的异常背景中断报警，导致测试数据彻底失效。

## 解决方案 (Solution)
1. **死锁 CPU 频率（Governor 设置）**：在跑测前通过 `echo performance | sudo tee /sys/devices/system/cpu/cpu*/cpufreq/scaling_governor` 强制锁定最高主频。这不仅抹平了由于 `ondemand` 降频带来的时间膨胀，还有效避免了测试启动瞬间 CPU 从深睡眠唤醒导致的剧烈波动。
2. **精准点杀幽灵进程**：弃用 `sudo [cmd] & PID=$! ; sudo kill -SIGINT $PID` 这种危险写法。经排查，该写法只能杀掉 `sudo` 这个 Wrapper，而将真正的 `loader` 探针进程变成了孤儿幽灵常驻后台，污染了后续的所有的测量环境。改为 `sudo killall -SIGINT loader` 强制清场。
3. **架构层解耦实证**：在严格隔离的环境下，最终获取了纯净数据，以实证强有力地证明了专利的核心论点——由于底层陷入机制同源，eBPF 与 KO 的**纯切入开销极度相似（~400ns）**；但由于 eBPF 的 BPF VM 及 Ringbuf 异步唤醒机制，其**负载开销（1000+ ns）** 远远大于原生 KO 的直接寻址消耗（~40 ns）。

## 避坑指南 (Pitfalls)
- **警惕伪相关的因果错觉**：当你看到某一项技术的开销数据异常高时（如 Native KO 跑出了 1600ns），**必须警惕环境未被隔离干净的前提**。本次排查中，通过自研的 `(/proc/interrupts)` 中断量伴随监控，敏锐地捕捉到了测试环境底噪被前一个测试污染的事实。
- **高频场景下的 eBPF 阿喀琉斯之踵**：[事实/证据] eBPF 的 `bpf_ringbuf_submit` 在极高频（百万次循环）场景下，会引发严重的中断风暴调度（高达 35 万次）。这意味着 eBPF 宣称的安全与高效，在特定“小而频”的追踪场景中会被其自身笨重的异步唤醒开销所反噬，这构成了工程落地中必须引入 `BPF_RB_NO_WAKEUP` 优化标志位的重要依据。
