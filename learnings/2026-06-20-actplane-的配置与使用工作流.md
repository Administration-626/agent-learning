# ActPlane 的配置与使用工作流

Date: 2026-06-20
Agent: Gemini CLI

## Background
了解了 ActPlane 的强大机制后，开发者需要掌握如何在实际开发中引入这套基于 eBPF 的内核级约束工具，以及它的配置作用域是如何管理的。

## Context / Problem
传统的系统级安全工具（如 AppArmor, SELinux）往往是全局的，配置极其复杂。但 AI Agent 的运行场景是多变的，不同的代码仓库需要不同的工作流约束（例如：生产级仓库严格限制访问路径，而个人实验仓库则需要较高的自由度）。如何平衡内核级控制的“重”与项目级开发的“轻”？

## Solution / Learnings
ActPlane 在设计上推荐采用**项目级（Per-Project）**的配置策略，提供了极简的 4 步工作流：

### 1. 作用域设计
核心配置文件 `actplane.yaml` 默认是项目级的。每个项目可以定义自己独立的底线规矩，实现了“轻量级沙箱”的体验。

### 2. 标准使用四步曲
1. **安装环境**：
   通过 Rust 官方仓库安装：`cargo install actplane`。
2. **初始化项目策略**：
   在具体的项目目录下执行 `actplane init`。这会在当前目录生成 `actplane.yaml`，开发者在此编写诸如“禁止 git push”或“必须经过测试”等定制化标签传播与时序规则。
3. **配置 Agent 传话筒 (Hook)**：
   内核虽然能拦截违规，但需要 Agent 框架配合读取反馈。需在项目配置中（如 `.claude/settings.local.json`）将工具调用失败（`PostToolUseFailure`）的事件指向命令：`actplane feedback-hook`。这一步架起了内核拦截与模型重推理之间的桥梁。
4. **受控启动 Agent**：
   不再直接敲击 `claude` 启动，而是必须包裹在 ActPlane 下：
   `sudo actplane run claude`

### 3. 安全权限的收放
虽然启动时必须带上 `sudo`，这是因为只有最高特权才能将 eBPF 探针注入系统内核。但注入操作一旦完成，ActPlane 会**自动降权**回普通用户，然后再拉起后续的 Agent 进程。安全隔离做得非常彻底。

## Pitfalls
- **裸奔启动**：配置了 yaml 策略，但忘记使用 `actplane run` 包装启动命令，导致约束形同虚设。
- **缺失反馈链**：只启动了 ActPlane 但忘了配置 Agent 框架的 Hook。此时当 Agent 违规时，它只会收到冰冷的系统报错而不知所措，失去了 Harness （导流）的意义，退化成了普通的死沙箱。
