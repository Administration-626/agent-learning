# AI Agent 权限控制：沙箱局限性与内核级约束

Date: 2026-06-20
Agent: Gemini CLI

## Background
在为 AI Agent 设置权限约束时，开发者和用户通常依赖 Prompt 提示词、工具层 Hook 拦截或者 Docker 沙箱。然而，在面对复杂的开发工作流约束时，这些常规手段往往会失效或表现糟糕。

## Context / Problem
现有三层约束机制各有其盲区：
1. **Prompt 约束的脆弱性**：Agent 容易在长上下文中遗忘规则，或产生非恶意的间接违规（例如：提示词要求不准删文件，但 Agent 运行了包含 `rm -rf` 的 Makefile target）。
2. **工具层 Hook 的视野盲区（API 拦截）**：Hook 只能看到表面传入的工具参数（如运行了 `python deploy.py`）。一旦 Agent 的脚本内部衍生了深层的子进程（如 `subprocess.run(["git", "push"])`），框架的 Hook 就会彻底“抓瞎”，无法阻止嵌套绕过。
3. **传统沙箱（如 Docker）的局限性**：
   - **缺乏时间概念（时序约束）**：沙箱只懂静态权限，无法表达“修改代码后，必须先跑测试才能 commit”这种业务逻辑。
   - **缺乏路径追溯（数据流约束）**：沙箱无法区分打开数据库文件的是合法的官方工具，还是 Agent 手写的“野脚本”。
   - **零反馈导致死循环**：被沙箱阻拦时，系统只返回冷冰冰的 `Permission denied`。Agent 无法理解被拒的业务原因，往往会陷入无限重试或幻觉死循环。

## Solution / Learnings
**ActPlane（基于 eBPF 的内核级约束引擎）**通过以下机制降维打击了这些盲区：
1. **下沉至必经之路（内核态）**：不管 Agent 怎么包装脚本、嵌套调用，只要读写文件或发起网络请求，最终必须产生系统调用（System Call）。eBPF 直接在最底层卡点，无视任何包装。
2. **标签传播（信息流追踪）**：给 Agent 主进程打上标签，该标签会自动传染给它创建的任何子进程（甚至通过文件传染）。规则依据标签执行，精准锁定恶意行为源头。
3. **时序推理（Since 子句）**：能够维护状态机，在事件时间线上做逻辑判断（如“在文件被写入后，是否发生过测试命令”）。
4. **建设性反馈（从 Sandbox 到 Harness）**：ActPlane 不只是强行阻断进程，更重要的是，它能在阻断的同时将具体的“违规原因”（如“禁止直接 push，请提示用户 review”）通过工具报错的 Hook 反馈给大模型。Agent 获取人类可读的提示后，能够立即自我纠偏，换合法路径完成任务。

## Pitfalls
- **沙箱万能论**：误以为把 Agent 关进 Docker 就万事大吉。沙箱防得了宿主机被破坏，防不住 Agent 在沙箱内部不按业务规矩出牌。
- **过度信任 Hook**：在业务安全层面，千万不要只在工具层 API 做参数校验，要深刻意识到“嵌套脚本”能轻易穿透工具层防护墙。
